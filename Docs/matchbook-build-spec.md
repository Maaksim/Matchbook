# Матчбук — Build Spec (v2)

Робоча назва: Матчбук (заміни на свою). Дитячий футбол: зберігач турнірів і матчів
як keepsake, де статистика — приємний бонус, а не суть. Головна відмінність від
конкурентів (STATZO): структура Турнір як окремий об'єкт + подача «спогад/
фотокнига», а не «дашборд/CV».

> **v2 — що змінилось відносно першої версії.** Цей файл замінює
> `matchbook-build-spec.pdf` як актуальне джерело правди (PDF лишається в
> репозиторії як історичний артефакт першої версії). Зміни зроблені за
> результатами технічного рев'ю: (1) виправлено неточне формулювання про
> сімейний CloudKit-шеринг у розділі 2; (2) додано вимогу до
> Repository-абстракції над CloudKit (розділ 3.9); (3) додано edit/delete
> флоу та порожні/помилкові стани (розділ 4.8); (4) уточнена поведінка
> paywall при downgrade (розділ 6); (5) додано розділ про offline-поведінку
> та синхронізацію (розділ 9); (6) додано accessibility-вимогу (розділ 7);
> (7) додано розділ відкритих питань (розділ 11).

---

## 1. Позиціювання (тримай перед очима під час розробки)

- Користувач: батько/мати дитини 6–14 років, що грає в турнірах.
- Робота, яку виконує застосунок: «зберегти назавжди турнір, який інакше
  забудеться».
- Емоція: гордість і ностальгія, а не аналітика.
- Наслідок для UX: фото на першому плані, великі трофейні картки, мінімум цифр
  на головних екранах. Це не таблиця — це альбом.

---

## 2. Технічний стек

- iOS 17+ (у нового застосунку немає причин тягнути iOS 16).
- SwiftUI + SwiftData (`@Model`, `@Query`, `@Environment(\.modelContext)`).
- **CloudKit (приватна база)** — безкоштовна синхронізація між пристроями
  **одного й того самого Apple ID** (наприклад, телефон і планшет одного з
  батьків). Це не те саме, що «обидва батьки бачать ту саму дитину» — якщо в
  мами і тата різні Apple ID, приватна база CloudKit між ними НЕ
  синхронізується. Перегляд одного профілю дитини з різних облікових записів
  — це окрема функція сімейного шерингу через `CKShare`, і в MVP вона не
  входить (див. розділ 5, Phase 2). Не обіцяй цю можливість користувачам до
  Phase 2.
- Без власного бекенда — але дивись розділ 3.9: доступ до даних все одно
  йде через Repository-протоколи, а не напряму з UI, саме на випадок, якщо
  рішення «без бекенда» доведеться переглянути (наприклад, через
  Android-версію чи потребу в реальному сімейному шерингу раніше Phase 2).
- StoreKit 2 для підписки/покупок.
- Стан: нативна Observation (`@Observable`), без `ObservableObject` /
  `@StateObject`. ViewModel заводимо тільки там, де логіка справді складна;
  більшість екранів працює прямо з `@Query`.
- Локалізація: String Catalog (`.xcstrings`) з першого дня. Базова мова —
  українська, друга — англійська (див. розділ 8).

## 3. Модель даних (SwiftData, готова до CloudKit)

Правила CloudKit, які тут дотримано: усі не-опціональні поля мають дефолти; усі
зв'язки опціональні; є зворотні зв'язки (`inverse`); немає `@Attribute(.unique)`.

*(Код моделей — без змін відносно v1, дивись `Matchbook-Technical-Documentation.md`
§3 для повного лістингу: `Player`, `Tournament`, `Match`, `MediaItem`,
`GoalMoment`.)*

### 3.9 Архітектура доступу до даних (Repository pattern) — **нова вимога**

Проблема, яку це вирішує: рішення «CloudKit, без бекенда» — правильне для
MVP, але це саме той тип рішення, який боляче міняти заднім числом, якщо
з'явиться потреба в Android-версії, реальному сімейному шерингу (не через
Apple-екосистему) чи server-driven логіці (remote config, A/B пейволу тощо).
Тому дані ізолюються від UI одразу, а не «коли знадобиться».

**Правило:** усі мутації (create/update/delete) і бізнес-агрегації йдуть через
Repository-протоколи, а не напряму з View/ViewModel у `modelContext`.

```swift
protocol PlayerRepository {
    func fetchAll() -> [Player]
    func create(_ player: Player) throws
    func update(_ player: Player) throws
    func delete(_ player: Player) throws
}

protocol TournamentRepository {
    func fetchAll(for player: Player) -> [Tournament]
    func create(_ tournament: Tournament, for player: Player) throws
    func update(_ tournament: Tournament) throws
    func delete(_ tournament: Tournament) throws   // каскадно видаляє Match/MediaItem
    func canCreateTournament(for player: Player, isSubscribed: Bool) -> Bool // paywall-гейт тут, не у View
}

protocol MatchRepository { /* аналогічно: create/update/delete/fetch */ }
protocol MediaRepository { /* create/delete + компресія фото/відео */ }
```

- **Реалізація за замовчуванням:** `SwiftDataPlayerRepository`,
  `SwiftDataTournamentRepository` тощо — обгортають `modelContext` і
  `@Query`-подібні фетчі.
- **Екрани зі списками** (`PlayerHomeView`, `TournamentDetailView`) можуть і
  далі використовувати `@Query` напряму для реактивного відображення — це
  нормальна SwiftData-ідіома, і повна відмова від неї заради «чистоти
  архітектури» суперечила б принципу простоти з розділу 2. Але *запис*,
  *видалення*, *paywall-перевірки* та *агрегації* (career stats, podiums
  тощо) мають жити в репозиторії, а не бути розкидані по View.
- **Що це дає:** якщо колись знадобиться Firebase (Firestore/Realtime DB) —
  міграція звужується до (1) нової реалізації тих самих протоколів
  (`FirestorePlayerRepository` і т.д.) і (2) заміни `@Query`-екранів на
  еквівалент через Firestore snapshot-listener, обгорнутий у
  `@Observable`. Бізнес-логіка (paywall-гейти, агрегації, валідація) не
  переписується взагалі, бо вона вже не прив'язана до SwiftData API.
- **Вартість:** трохи більше boilerplate на старті (протокол + одна
  реалізація на кожен агрегат). Це свідомий trade-off заради опційності,
  а не обов'язкова повна мінімізація складності.

## 4. Фічі MVP (v1) — готові до реалізації

Формат кожної фічі: що робить → екрани → ключова логіка → дані → критерії
готовності. Пиши AI по одній фічі за раз.

### 4.1 Профіль дитини (Player)
- Що: створити/редагувати/**видалити** дитину; підтримка кількох дітей
  (перемикач зверху).
- Екрани: список дітей (якщо >1) → форма (ім'я, фото з PhotosPicker, номер,
  позиція, клуб, дата народження).
- Логіка: `@Query` усіх Player, сортування за createdAt. Активна дитина — у
  `@AppStorage("activePlayerID")`. Create/update/delete — через
  `PlayerRepository` (розділ 3.9).
- **Видалення:** видалення дитини каскадно видаляє всі її турніри/матчі/фото
  (`.cascade` у моделі). Обов'язковий confirmation-діалог з явним
  попередженням про кількість турнірів/фото, які будуть втрачені
  («Видалити Марка та 4 турніри, 87 фото? Це незворотно»). Якщо активна
  дитина видаляється — активним стає наступний Player у списку, або
  показується empty state з розділу 4.8, якщо дітей не лишилось.
- Дані: `Player`.
- Готово коли: можна створити дитину з фото, вона стає активною, дані
  переживають перезапуск; видалення працює з підтвердженням і без крашу при
  видаленні активної дитини.

### 4.2 Створення турніру (Tournament)
- Що: завести/редагувати/**видалити** турнір під активну дитину.
- Екрани: форма (назва, дати від/до, місто, майданчик, формат, назва
  команди, обкладинка-фото).
- Логіка: новий Tournament, `player = activePlayer`, через
  `TournamentRepository`. Перед створенням — виклик
  `canCreateTournament(for:isSubscribed:)` для free-ліміту (розділ 6).
- **Видалення:** confirmation-діалог («Видалити турнір і всі 6 матчів, 40
  фото?»). Каскадно видаляє Match + MediaItem цього турніру.
- Дані: `Tournament`.
- Готово коли: турнір з'являється у списку турнірів дитини, відсортований за
  датою (нові зверху); видалення і редагування працюють з підтвердженням.

### 4.3 Швидке додавання матчу (Match) — головний потік
- Що: внести/редагувати/**видалити** матч у 3 тапи прямо на бровці.
- Екран: одна форма-лист: суперник (текст) → рахунок (два степери
  teamScore/opponentScore) → голи/асисти дитини (степери). Все інше (стадія,
  хвилини, оцінка, MOTM, нотатка) — під `DisclosureGroup` «Деталі».
- Логіка: прив'язка до поточного турніру; `outcome` рахується сам; кнопка
  «Зберегти й додати ще» для турнірів на кілька ігор поспіль. Запис — через
  `MatchRepository`.
- **Офлайн:** збереження матчу — це локальний SwiftData-запис, він
  спрацьовує миттєво незалежно від інтернету на стадіоні; синхронізація в
  CloudKit відбувається у фоні, коли з'явиться мережа (див. розділ 9).
- **Видалення:** свайп або кнопка в деталях матчу, з коротким
  підтвердженням (без важкого діалогу — матч менш «дорога» одиниця, ніж
  турнір, але випадкове видалення все одно має підтверджуватись).
- Дані: `Match`.
- Готово коли: матч додається < 15 секунд, зберігання не блокує UI (і не
  вимагає інтернету), «додати ще» лишає контекст турніру, видалення й
  редагування вже збереженого матчу працюють.

### 4.4 Підсумок турніру (placement + трофей)
- Що: зафіксувати фінальне місце й отримати «трофейну» картку.
- Екран: деталь турніру: зверху героїчний блок (обкладинка + місце + медаль-
  емодзі + placementLabel), нижче — список матчів, під ним — агрегати (В-
  Н-П, голи, асисти) дрібним текстом.
- Логіка: вибір місця з пресетів (Чемпіон / Фіналіст / 3-тє / Груповий етап /
  Без місця); finalPlacement + placementLabel синхронні.
- Дані: `Tournament`.
- Готово коли: подіумний турнір візуально виділяється, картка виглядає як
  нагорода.

### 4.5 Медіа (фото)
- Що: додати/**видалити** фото до матчу і до турніру.
- Екран: сітка фото на деталі матчу/турніру; додавання через PhotosPicker
  (мультивибір); видалення — long-press → «Видалити», з підтвердженням.
- Логіка: зберігати стиснене (макс. ~1600px, JPEG ~0.7) у `MediaItem.data`
  (external storage), через `MediaRepository`. Показувати thumbnail-и,
  повне — по тапу.
- Дані: `MediaItem`.
- Готово коли: 20+ фото не гальмують і не роздувають БД (перевір, що йдуть у
  external storage); видалення окремого фото не займає весь турнір.

### 4.6 Картка для шерингу (вірусний гачок №1)
- Що: згенерувати гарну картку матчу/турніру для чатів команди й сторіз.
- Екран: прев'ю картки + `ShareLink`. Рендер SwiftUI-в'юхи в Image через
  `ImageRenderer`.
- Контент картки: фото дитини/турніру, ім'я, місце/медаль, рахунок або
  підсумок «5 ігор • 4 голи». Ватермарк-логотип застосунку (це і є твій
  органічний маркетинг).
- **Приватність:** картка містить ім'я і фото дитини і призначена для
  шерингу поза застосунком (у месенджери). Явно позначай у прев'ю, що саме
  потрапить на картку, перш ніж дати `ShareLink` — без прихованих полів
  (наприклад, не додавай геолокацію/точну адресу майданчика на картку без
  окремої згоди).
- Логіка: `ImageRenderer(content:).uiImage`, шер як PNG.
- Готово коли: картка читабельна в квадраті й 9:16, ватермарк присутній.

### 4.7 Кар'єрний огляд дитини
- Що: один екран-«альбом усього шляху».
- Екран: шапка з агрегатами (турнірів, матчів, голів, асистів, подіумів) +
  таймлайн турнірів картками (обкладинка + місце).
- Логіка: обчислювані властивості Player (або агрегація в
  `PlayerRepository`, якщо стає важкою для computed property).
- Готово коли: таймлайн гортається від першого турніру, подіуми помітні.

### 4.8 Порожні та помилкові стани — **нова вимога**

Специфіковані раніше лише «перший запуск». Цього недостатньо для
sync-залежного застосунку:

- **iCloud не налаштований / вимкнений на пристрої.** Застосунок все одно
  має повністю працювати локально (SwiftData без CloudKit-дзеркалення
  просто не синкає). Показати ненав'язливий банер один раз («Дані
  зберігаються тільки на цьому пристрої — увімкни iCloud, щоб не втратити їх
  при заміні телефону»), не блокувати flow.
- **Новий пристрій, синк ще не завершився.** Перший запуск на другому
  пристрої того самого Apple ID може на кілька секунд/хвилин показати
  порожній стан, поки CloudKit не підтягнув дані. Показуй короткий
  «Синхронізуємо…» індикатор замість welcome-екрана «Додати дитину», якщо
  `NSUbiquitousKeyValueStore`/CloudKit-контейнер вказує, що акаунт
  налаштований і, ймовірно, вже має дані (щоб не підштовхнути користувача
  випадково створити дубль дитини).
- **Сховище iCloud заповнене.** Синк фото/відео може тихо не проходити.
  Раз на сесію перевіряй статус останньої синхронізації й показуй
  ненав'язливе попередження, якщо запис у CloudKit постійно фейлиться
  (не блокуючий toast/banner, не alert на кожному екрані).
- **Турнір без жодного матчу.** Показати короткий заклик «Додай перший
  матч» замість порожнього списку.
- **Видалення з каскадом.** Кожна операція видалення з дочірніми об'єктами
  (Player → Tournament → Match/MediaItem) вимагає підтвердження з коротким
  описом обсягу втрати (див. 4.1–4.3).

## 5. Фічі Phase 2 (після валідації MVP)

- Моменти голів (`GoalMoment`) — тип голу, хвилина, відео до конкретного
  голу. Замінює прості лічильники.
- Фотокнига турніру/сезону — генерація PDF (обкладинка, матчі, фото,
  підсумок) з можливістю зберегти/надрукувати. Головний платний гачок.
- Віджети / Lock Screen — кар'єрні цифри дитини, найближчий турнір.
- «Цього дня рік тому» — локальні нотифікації-спогади (утримання поза
  сезоном). Реалізується через `UNUserNotificationCenter` локально, сервер
  не потрібен.
- Бейджі/віхи — перший гол, хет-трик, перший трофей.
- **Приватний сімейний шеринг (`CKShare`)** — бабуся/тато з іншим Apple ID
  бачать той самий профіль дитини (тільки перегляд, або перегляд+запис —
  визначити при плануванні фічі). Це та функція, про яку розділ 2 раніше
  помилково натякав як про частину базового CloudKit-синку — насправді
  вона вимагає окремої реалізації через `CKShare`/`UICloudSharingController`
  і належить сюди.
- Друковані картки-стилі — кілька преміум-шаблонів шер-карток.

## 6. Монетизація (StoreKit 2) — не копіюй жорсткий пейвол STATZO

STATZO закриває все після 1 матчу — це вбиває звичку й органіку, а тобі саме
батьківські чати дають ріст.

**Безкоштовно** (щоб сформувалась звичка й розійшлись картки):
- 1–2 турніри повністю, необмежені матчі в них.
- Базова шер-картка (з ватермарком).
- 1 дитина.

**Матчбук+** (підписка ~$3–5/міс або річна зі знижкою):
- Необмежені турніри.
- Кілька дітей.
- Необмежені фото/відео.
- Преміум-стилі карток без ватермарка.
- Віджети + спогади.

**Разова покупка** (окремо від підписки):
- Фотокнига турніру/сезону (PDF/друк) — $5–15 за штуку. Батьки платять за
  keepsake охочіше, ніж за графіки; це монетизує навіть тих, хто не хоче
  підписки.

**Downgrade-поведінка — нова вимога.** Коли підписка Matchbook+
закінчується (не поновлюється), турніри/діти понад free-ліміт **не
ховаються і не видаляються** — вони переходять у **read-only**: видно у
списках, фото/матчі відкриваються на перегляд, але не можна додати новий
матч/фото/турнір понад ліміт, поки підписка неактивна. Це узгоджується з
позиціюванням «людина платить, щоб зберегти назавжди» — застосунок ніколи
не повинен виглядати так, ніби він «забрав» дані, за які вже заплачено чи
які вже введені.

**Restore Purchases — нова вимога.** StoreKit 2 flow повинен явно включати
кнопку/дію «Відновити покупки» на екрані пейволу (обов'язкова вимога App
Review; актуальна також при переустановці/зміні пристрою).

**Логіка:** людина платить не «щоб продовжити користуватись», а «щоб
зберегти назавжди».

---

## 7. Дизайн-напрям (щоб не вийшов черговий дашборд)

- Тепла, «альбомна» естетика: багато фото, великі заокруглення
  (`clipShape(.rect(cornerRadius:))`), м'які тіні.
- Подіумні турніри — акцентні кольори золота/срібла/бронзи.
- Мінімум таблиць на головних екранах; цифри — дрібним, вторинним планом.
- Своя палітра в Assets, `foregroundStyle`, кнопки замість `onTapGesture`,
  `@ViewBuilder` замість `AnyView`.
- **Accessibility — нова вимога.** Підтримка Dynamic Type (текст не має
  ламати layout на найбільших розмірах шрифту); VoiceOver-labels для
  іконок-only контролів (степери, floating action button, share-іконка);
  мінімальний контраст тексту на золотому/зеленому фоні перевірити окремо
  (декоративні золоті бейджі — це радше вторинний акцент, не єдиний носій
  інформації). Не блокує MVP-функціональність, але має бути в чеклісті
  перед релізом, не «колись потім».

## 8. Локалізація

Коротка відповідь: так, сенс є — але цінність не там, де здається. Дитячий/
аматорський футбол універсальний (STATZO зібрав завантаження в 20+ країнах),
а твій застосунок не залежить від зовнішніх футбольних даних — усе вводить
користувач. Тому він працює в будь-якій країні «з коробки»: щоб вийти на
новий ринок, потрібен лише переклад інтерфейсу, а не збір локальної бази
ліг.

Що реально дає ROI:
1. Не зашивати рядки в код від першого дня — вже закладено через
   `LocalizedStringResource`.
2. Локалізувати метадані App Store (назва, підзаголовок, ключові слова,
   скріншоти) під кожен регіон.
3. Сам переклад інтерфейсу на 1–3 мови на старті. Більше — після валідації.

Мови: старт — українська + англійська; швидкі наступні — польська,
іспанська, португальська, німецька.

**Технічний підхід (SwiftUI, iOS 17+):** String Catalog (`.xcstrings`);
`Text("...")` / `String(localized:)` локалізуються автоматично; множина
критична для української (one/few/many) — тільки через plural-ключі, ніколи
конкатенацією; дати/числа — тільки `Date.FormatStyle`/`.formatted(...)`;
RTL-готовність через `leading`/`trailing`.

**Локалізувати:** весь інтерфейс, назви позицій/форматів, підписи, кнопки,
згенеровані рядки (підсумок турніру, текст шер-картки), з коректною
множиною.

**НЕ локалізувати:** контент користувача — ім'я дитини, суперники, назви
турнірів, нотатки, фото.

## 9. Offline-поведінка та синхронізація — **новий розділ**

- **Локальний запис завжди першочерговий.** Усі створення/редагування
  (особливо швидкий матч, розділ 4.3) пишуться в локальну SwiftData-базу
  синхронно й миттєво, незалежно від наявності мережі. Це критично, бо
  основний use case — стадіон/спортивний майданчик з нестабільним
  зв'язком.
- **CloudKit-синк — фоновий і best-effort.** SwiftData сам мірить зміни у
  приватну CloudKit-базу, коли з'являється мережа. UI не повинен
  блокуватись і не повинен показувати помилку, якщо синк відкладається —
  тільки якщо він падає систематично (розділ 4.8).
- **Конфлікти запису.** Якщо два пристрої одного Apple ID редагують той
  самий об'єкт офлайн одночасно (рідкісний, але можливий кейс — наприклад,
  телефон і планшет одного з батьків), CloudKit/SwiftData застосовує
  стандартне злиття на рівні полів (останній запис перемагає для
  зміненого поля). **Дублікати записів** (наприклад, той самий матч,
  внесений двічі різними пристроями до синку) — це прийнятий ліміт MVP,
  явно не вирішується. Якщо стане реальною проблемою в проді — розглянути
  дедуплікацію за (opponent + date + tournament) при виявленні синку.
- Це чергове місце, де Repository-абстракція (розділ 3.9) корисна: вся
  логіка «зберегти зараз, синкнути пізніше» інкапсульована в реалізації
  репозиторію, а не розмазана по View.

## 11. Промпти для AI-реалізації (копіюй по черзі)

1. **Каркас проєкту:** «Створи новий SwiftUI-застосунок iOS 17+ з SwiftData
   та CloudKit. Додай моделі з розділу 3 цього спеку. Створи
   Repository-протоколи з розділу 3.9 (`PlayerRepository`,
   `TournamentRepository`, `MatchRepository`, `MediaRepository`) і їхні
   SwiftData-реалізації. Налаштуй `modelContainer` з CloudKit. Зроби
   `RootView` з перемикачем активної дитини через `@AppStorage`.»
2. **Профіль + турніри (4.1–4.2):** «Реалізуй `PlayerEditView` (PhotosPicker
   для аватара) з create/edit/delete через `PlayerRepository`, і
   `PlayerHomeView` зі списком турнірів активної дитини через `@Query`.
   Додай `TournamentEditView` з create/edit/delete через
   `TournamentRepository`, включно з `canCreateTournament` paywall-гейтом.
   Кожне видалення — з підтвердженням, що показує обсяг каскаду. Дотримуйся:
   `@State private`, `foregroundStyle`, `clipShape(.rect(cornerRadius:))`,
   `Button` замість `onTapGesture`.»
3. **Швидкий матч (4.3):** «Зроби `MatchEditView` — форма додавання/
   редагування/видалення матчу зі степерами рахунку й голів/асистів, деталі
   під `DisclosureGroup`, кнопка «Зберегти й додати ще». Прив'яжи до
   переданого Tournament через `MatchRepository`. Переконайся, що збереження
   не залежить від мережі (локальний SwiftData-запис).»
4. **Герой турніру + медіа (4.4–4.5):** «Зроби `TournamentDetailView`:
   героїчний блок з обкладинкою, місцем і медаллю; список матчів; сітку
   фото з мультивибором PhotosPicker і видаленням окремого фото; фото
   стискай до 1600px перед збереженням у `MediaItem` через
   `MediaRepository`.»
5. **Шер-картка (4.6):** «Зроби `ShareCardView` — квадрат і 9:16, з фото,
   ім'ям, підсумком і ватермарком. Рендер через `ImageRenderer`, віддай у
   `ShareLink` як PNG. Прев'ю перед шерингом має явно показувати, що саме
   потрапить на картку.»
6. **Порожні/помилкові стани (4.8):** «Додай стани: немає iCloud-акаунта
   (банер, локальна робота), синк у процесі на новому пристрої (індикатор
   замість welcome-екрана), турнір без матчів (заклик додати перший матч),
   confirmation-діалоги для кожного каскадного видалення.»
7. **Пейвол (розділ 6):** «Додай StoreKit 2: підписку Матчбук+ і разову
   покупку «Фотокнига». Ліміт безкоштовного тарифу — 2 турніри; при спробі
   створити третій показуй пейвол через `canCreateTournament`. Реалізуй
   downgrade у read-only (не приховування/видалення) для контенту понад
   ліміт після закінчення підписки. Додай кнопку «Відновити покупки».»
8. **Локалізація (розділ 8):** «Увімкни String Catalog (`.xcstrings`),
   базова мова українська, додай англійську. Перевір, що всі UI-рядки
   локалізовані, а не зашиті. Число голів/матчів/турнірів виводь через
   plural-ключі (one/few/many), а не конкатенацією. Дати й числа форматуй
   через `Date.FormatStyle`/`.formatted()`. Згенерований текст шер-картки
   зроби локалізованим і з коректною множиною.»

## 12. Відкриті питання — **новий розділ**

Свідомо не вирішені зараз, щоб не блокувати старт розробки — але мають
з'явитись у беклозі до релізу:

- **Crash reporting / analytics.** Рішення не прийняте (Sentry/TelemetryDeck
  проти суто Apple-інструментів (Xcode Organizer + MetricKit) проти взагалі
  нічого на MVP). Питання приватності (дані про дітей) варто врахувати при
  виборі — мінімізація сторонніх SDK є свідомою перевагою поточної
  архітектури, і будь-який вибір тут має це зважити.
- **Android-версія.** Наразі не планується, тому CloudKit-only лишається в
  силі. Але це рішення варто периодично переглядати: якщо Android
  з'явиться в роадмапі, Repository-абстракція (розділ 3.9) звужує вартість
  міграції, але не усуває її — `@Query`-екрани й сам CloudKit-шар все одно
  доведеться замінювати.
- **Валідаційні правила полів** (мін/макс shirtNumber, від'ємний рахунок
  матчу тощо) — не специфіковані, залишені на розсуд імплементації First
  MVP-проходу.
